<task type="auto">
  <name>Task 2: Create cluster metrics router with utilization endpoint</name>
  <files>job_monitor/backend/routers/cluster_metrics.py</files>
  <action>
Create new router file `cluster_metrics.py`:

1. Router setup:
   - `router = APIRouter(prefix="/api/cluster-metrics", tags=["cluster-metrics"])`

2. `GET /api/cluster-metrics/{job_id}` endpoint:
   - Response model: ClusterUtilization
   - Query parameter: `runs: int = Query(default=5, ge=1, le=10)` for number of runs to analyze
   - Dependencies: `ws = Depends(get_ws)`, `warehouse_id = Depends(get_warehouse_id)`

3. Implementation approach (from RESEARCH.md - proxy calculation):
   - Query billing data for DBU consumption patterns across recent runs
   - Calculate utilization proxy: DBUs per hour normalized against cluster capacity
   - For each run, get cluster_id from job run, look up node types
   - Estimate utilization based on DBU consumption rate vs expected capacity

   SQL query pattern (use asyncio.to_thread for execute_statement):
   ```sql
   WITH job_runs AS (
       SELECT run_id, job_id, cluster_id, run_duration_seconds, period_start_time
       FROM system.lakeflow.job_run_timeline
       WHERE job_id = :job_id
           AND period_start_time >= current_date() - INTERVAL 30 DAYS
       ORDER BY period_start_time DESC
       LIMIT :runs
   ),
   billing AS (
       SELECT usage_metadata.job_id, cluster_id, usage_date,
              SUM(usage_quantity) as total_dbus
       FROM system.billing.usage
       WHERE usage_metadata.job_id = :job_id
       GROUP BY 1, 2, 3
       HAVING SUM(usage_quantity) != 0
   )
   SELECT jr.run_id, jr.run_duration_seconds, b.total_dbus,
          (b.total_dbus / NULLIF(jr.run_duration_seconds / 3600.0, 0)) as dbus_per_hour
   FROM job_runs jr
   LEFT JOIN billing b ON DATE(jr.period_start_time) = b.usage_date
   ```

4. Utilization calculation logic:
   - Normalize DBUs/hour to percentage (use typical node type DBU rates as baseline)
   - Apply heuristic: <1 DBU/hour = ~20%, 1-2 = ~40%, 2-4 = ~60%, >4 = ~80%
   - Split roughly between driver (30%) and workers (70%) for estimation
   - Mark as over-provisioned if ALL of last 5 runs have <40% average utilization

5. Right-sizing recommendation logic:
   - If over-provisioned and avg utilization <20%: "Consider reducing workers by 50%"
   - If over-provisioned and avg utilization 20-30%: "Consider reducing to {n-1} workers"
   - If over-provisioned and avg utilization 30-40%: "Consider using smaller node types"
   - Recommendation stored as plain text string

Note: This is proxy-based estimation since actual CPU/memory metrics aren't in system tables. The UI will show "Estimated" label to indicate this.

   ```sql
   ORDER BY ordinal_position
   ```

4. Schema comparison logic (from RESEARCH.md):
   ```python
   def detect_schema_drift(previous: list[dict], current: list[dict]) -> SchemaDrift:
       prev_cols = {c["column_name"]: c for c in previous}
       curr_cols = {c["column_name"]: c for c in current}

       added = list(set(curr_cols.keys()) - set(prev_cols.keys()))
       removed = list(set(prev_cols.keys()) - set(curr_cols.keys()))

       type_changes = []
       for col in set(prev_cols.keys()) & set(curr_cols.keys()):
           if prev_cols[col]["data_type"] != curr_cols[col]["data_type"]:
               type_changes.append(ColumnChange(
                   column_name=col,
                   change_type="type_changed",
                   old_type=prev_cols[col]["data_type"],
                   new_type=curr_cols[col]["data_type"]
               ))

       return SchemaDrift(
           has_drift=bool(added or removed or type_changes),
           added_columns=added,
           removed_columns=removed,
           type_changes=type_changes
       )
   ```

5. Job-to-table mapping:

   - If not found, return empty list with informative message
   - Future: Could integrate with system.access.table_lineage if available

Note: Row count from information_schema may be approximate. Document this limitation.
