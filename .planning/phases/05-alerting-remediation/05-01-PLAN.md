Create alerts.py router that generates alerts dynamically from existing data sources:

1. In-memory acknowledgment store at module level:
   ```python
   _acknowledged: dict[str, datetime] = {}  # condition_key -> acknowledged_at
   ```

2. Helper functions:
   - is_acknowledged(condition_key: str) -> bool: Check if acknowledged within 24h TTL
   - _generate_failure_remediation(job, failure_reasons: list[str]) -> str: Context-aware suggestion
   - _generate_sla_remediation(job, elapsed_pct: float, sla_minutes: int) -> str
   - _generate_cost_remediation(anomaly_type: str, multiplier: float, baseline: float) -> str
   - _generate_cluster_remediation(utilization: float, runs_analyzed: int) -> str

3. GET /api/alerts endpoint with optional query params:
   - severity: list[str] | None (filter by P1, P2, P3)
   - category: list[str] | None (filter by failure, sla, cost, cluster)
   - acknowledged: bool | None (filter by acknowledged status)

   Alert generation logic (call existing endpoints internally via direct function calls, not HTTP):

   a) Failure alerts from health_metrics:
      - Call existing health metrics logic for 7-day window
      - P1: Jobs with priority="P1" (2+ consecutive failures)
      - P2: Jobs with priority="P2" (single failure)
      - P3: Jobs with priority="P3" (yellow zone 70-89%)
      - Remediation: Check failure_reasons for OOM/timeout patterns

   b) SLA alerts:
      - Query running jobs via Jobs API (jobs_api.py endpoint)
      - For each running job with SLA target (from job tags):
        - Calculate elapsed_pct = (current_duration / sla_seconds) * 100
        - P1: elapsed_pct >= 100 (actual breach)
        - P2: elapsed_pct >= 80 (breach risk)
      - Remediation: Include SLA target and remaining time

   c) Cost alerts:
      - Call existing cost anomalies logic
      - P2: cost_spike anomalies (>2x baseline)
      - For budget alerts, check job tags for budget_monthly_dbus:
        - Query current month's usage for jobs with budget tags
        - P2: usage >= 80% of budget (approaching)
        - P1: usage >= 100% of budget (exceeded)
      - Remediation: Include specific multiplier and baseline values

   d) Cluster alerts:
      - Get over-provisioned jobs from cluster_metrics
      - P3: Jobs flagged as over_provisioned with <40% utilization
      - Remediation: Include specific worker reduction suggestion

   Return AlertListOut with alerts sorted by severity (P1 first, then P2, then P3) and by_severity counts.

4. POST /api/alerts/{alert_id}/acknowledge endpoint:
   - Extract condition_key from alert_id
   - Store in _acknowledged dict with current timestamp
   - Return updated Alert with acknowledged=True

5. Register router in app.py with prefix="/api/alerts"

Important implementation notes:
- Use asyncio.gather for parallel data fetching from multiple sources
- Apply acknowledgment status from _acknowledged store to generated alerts
- Deduplicate by condition_key (higher severity wins if same condition)
- SLA breach risk only for jobs with result_state IS NULL (still running)
- Follow existing pattern: use ws=Depends(get_ws) for WorkspaceClient
